# AlphaNexus 后续实施方案

版本：v1.0  
日期：2026-03-25  
适用阶段：完成首轮项目启动与基础并行开发之后  
目标：把 AlphaNexus 从“可启动原型”推进到“可日常使用的 MVP”

---

## 1. 文档目的

这份文档回答的是：

- 首轮 5 个并行线程完成并集成后，下一步怎么推进
- 哪些工作要先做，哪些可以后做
- 如何避免项目进入“功能很多但主流程不顺”的状态
- 怎么从原型走到可实际盯盘使用的 MVP

核心原则：

> 后续实施必须围绕一条真实可用的纵向链路推进，而不是横向堆功能。

这条纵向链路是：

`打开 Session -> 截图 -> 标注 -> 记观点 -> 发 AI -> 记录开仓/离场 -> 自动生成复盘草稿 -> 导出笔记`

只要这条链路顺，AlphaNexus 就已经具备实战价值。

---

## 2. 当前阶段定义

假设 Kickoff 之后，当前系统已经具备：

- Electron 桌面壳
- React 页面骨架
- SQLite 初始化
- SessionWorkbench 页面
- mock 事件流
- Capture/Annotation demo
- AI provider 骨架
- Markdown 导出骨架

这时项目所处阶段不是“完成”，而是：

**Stage 0：可演示原型**

这个阶段的目标只是证明架构方向正确，还不能直接高频日常使用。

接下来应进入：

**Stage 1：可连续使用的 MVP 开发**

---

## 3. 后续实施总路线

我建议后续按 5 个阶段推进：

1. 基础链路打通
2. 真实数据落地
3. 交易线程闭环
4. 周/月聚合成型
5. 稳定性、体验和 AI 质量强化

不要跳步骤。

尤其不要在基础链路没顺之前，就先做：

- 复杂周报
- 高级统计图
- 多 AI 共识系统
- 自动概率校准

这些都应该排在主流程稳定之后。

## 4. 阶段一：基础链路打通

### 4.1 阶段目标

把原型拉到一个真正可用的最小闭环：

- 真正创建 Session
- 真正创建 Event
- 真正保存截图与标注
- 真正保存笔记块
- 真正发起 AI 请求
- 真正把结果写回事件流

### 4.2 必做事项

#### A. Session 与 Current Context 真正落地

必须从 mock 进入真实状态管理：

- 当前合约
- 当前 Session
- 当前 Trade
- 当前挂载目标

并实现：

- 新建 Session
- 继续 Session
- 切换当前上下文

#### B. CaptureResult 真正落盘

把截图和标注从 demo 输出，变成真正的持久化资产：

- 原图保存
- 标注图保存
- 标注 JSON 保存
- 关联 Event 和 Screenshot 记录

#### C. 右侧工作台真正可编辑

不再只是 UI 占位，要实现：

- 新建文本块
- 自动保存
- 改挂载目标
- 删除与撤销

#### D. AI 调用打通一条真链路

先只接一个 provider，建议 OpenAI 或你已有最稳定的 API。

要求：

- 发送图片和上下文
- 返回结构化结果
- 结果写入 ai_runs 与 analysis_cards
- 在 UI 中显示摘要卡和深度分析

#### E. Markdown 导出可真正使用

至少导出：

- Session 标题
- 时间序列
- 图片引用
- 用户观点
- AI 摘要

### 4.3 阶段验收标准

满足以下 8 项才算通过：

1. 能新建一个真实 Session
2. 能截图并保存到本地
3. 能在截图上做标注并保存
4. 能添加一条“我的实时看法”
5. 能将截图和观点发给 AI
6. 能把 AI 摘要写回事件流
7. 能记录一次开仓和一次离场
8. 能导出一个完整的 Session Markdown

### 4.4 推荐并行拆分

- 线程 A：Current Context + Session 状态管理
- 线程 B：CaptureResult 存储链路
- 线程 C：右侧工作台可编辑块
- 线程 D：AI 真调用与结果入库
- 线程 E：Markdown 导出打通

## 5. 阶段二：真实数据落地

从“能用一次”提升到“能连续记录一整段盯盘过程”。

### 5.2 必做事项

#### A. 事件流从 mock 改为数据库驱动

能力要求：

- 实时刷新
- 按时间排序
- 按类型筛选

#### B. 删除、恢复、移动真正生效

你明确要求：

- 删图
- 删 AI
- 删任意层级笔记

因此这一阶段必须完成：

- 内容块软删除
- 回收站
- 撤销
- 移动到其他 Session / Trade

#### C. 目标选择器可用

必须能把内容挂到：

- 当前 Session
- 当前 Trade
- 历史 Session
- 历史 Trade

#### D. 图片查看与多图对照

至少支持：

- 单图查看
- 多图 tabs
- Setup / Exit 对照

### 5.3 阶段验收标准

1. 一天内可连续创建 20 条以上事件不卡壳
2. 删除/恢复任意块不会破坏事件流
3. 可把截图从当前主线移动到某笔历史 Trade

### 5.4 推荐并行拆分

- 线程 A：Timeline 真实数据接入
- 线程 B：回收站、删除、撤销
- 线程 C：目标选择器与挂载迁移
- 线程 D：多图查看与对照

## 5A. 阶段二点五：知识库基础与上下文输入层

### 5A.1 阶段目标

把 AlphaNexus 从“能记录事件”推进到“能带着方法论记录事件”。

这一阶段建立两层能力：

- 知识库基础链路
- 上下文感知 Composer 基础链路

其顺序必须明确：

`资料导入 -> 抽取草稿 -> 审核发布 -> approved 知识卡 -> 运行时检索 -> Composer / 标注评估 / 锚点绑定`

### 5A.2 必做事项

#### A. 知识库最小闭环

至少实现：

- 导入资料
- 文档分块
- 草稿知识卡入库
- 审核并发布为 approved
- 基于合约 / 周期 / 标签的最小检索

#### B. Composer 基础版

至少实现：

- 右侧候选 Chips
- 结构化模板插入
- 规则与知识卡驱动候选
- 已接受候选记录

#### C. 标注语义化

标注至少要支持：

- 标题
- 语义类型
- 备注
- 是否加入记忆

#### D. AI 候选标注最小链路

至少实现：

- 候选标注 schema
- 画布候选层
- 保留 / 丢弃

### 5A.3 阶段验收标准

1. 至少一份资料可导入并发布为 approved 知识卡
2. SessionWorkbench 能基于当前上下文给出候选短语或模板
3. AI 分析时能带入相关知识卡摘要
4. 用户可把一个正式标注升级为记忆候选

### 5A.4 推荐并行拆分

- 线程 A：知识导入与审核
- 线程 B：知识检索与 runtime 接口
- 线程 C：Composer 基础 UI
- 线程 D：标注语义化与候选标注层

### 5A.5 兼容与迁移说明

- `annotations` 增加语义字段：`title`、`semantic_type`、`note_md`、`add_to_memory`。
- 数据迁移采用增量方式：旧行回填为 `title = label`、`note_md = text`、`add_to_memory = 0`，`semantic_type` 保持 `NULL`，因此历史截图与历史标注仍可直接读取。
- Capture 标注输入保持向后兼容：旧 payload 即使没有携带这些新增字段，保存时也会自动补默认值，不会阻断截图、标注和事件落盘。
- 知识审核发布新增 `edit-approve` 动作：允许审核时直接修订草稿并发布为 `approved`；旧的 `approve` 仍然兼容。
- runtime 检索和 Composer 运行时上下文只读取 `approved` 知识卡；`draft` 只停留在导入与审核链路，不进入盘中主记录流程。

## 6. 阶段三：交易线程闭环

### 6.1 阶段目标

把“事件流记录”升级为“完整交易闭环”。

这一阶段完成后，AlphaNexus 才真正开始像交易复盘工具，而不是高级笔记软件。

### 6.2 必做事项

#### A. Trade 创建与状态流转

需要明确支持：

- Open
- Add
- Reduce
- Exit
- Cancel

#### B. TradeDetail 页面完善

最少要具备：

- Setup 图
- Manage 图
- Exit 图
- 我的计划
- AI 当时建议
- 实际执行
- 事后复盘

#### C. 离场截图自动引导

这是你的关键需求之一。

离场时必须支持：

- 快速截图
- 自动建议挂到当前 open trade
- 一键作为 Exit 图保存
- 自动生成结果评估草稿

#### D. 交易级 AI 复盘

输入：

- Setup 图
- Exit 图
- 计划
- 执行
- 结果

输出：

- 做得好的地方
- 出错点
- 下次改进

### 6.3 阶段验收标准

1. 一笔交易能从开仓追踪到离场
2. Exit 图可以和 Setup 图形成完整对照
3. 可查看该笔交易的 AI 观点与人工观点
4. 能生成交易级复盘卡

### 6.4 推荐并行拆分

- 线程 A：Trade 状态机与数据层
- 线程 B：TradeDetail 页面
- 线程 C：Exit 图快捷流程
- 线程 D：交易级 AI 复盘 prompt 与输出

### 6.5 兼容与迁移说明

- 本阶段合同变更均为增量扩展：
  - `trade.status` 新增 `canceled`
  - `event.event_type` 新增 `trade_cancel`
  - `TradeDetailPayload` 新增 `ai_groups`
  - `RunAiAnalysisInput` / `RunMockAiAnalysisInput` 新增可选 `trade_id`
  - `SavePendingSnipResult` 新增可选 `resolved_target`
- 兼容性影响：
  - 旧 Trade 行的 `planned` / `open` / `closed` 状态保持不变，读取逻辑向后兼容。
  - 旧 Event 行不需要回填 `trade_cancel`，仅新产生的取消事件会使用该类型。
  - 旧的 TradeDetail 调用方如果只消费 `latest_analysis_card` / `linked_ai_cards` 仍可继续工作；新增 `ai_groups` 只用于显式区分盘中分析与交易复盘。
  - 旧 Capture 保存结果如果没有 `resolved_target`，前端仍按基础保存成功逻辑处理。
- 迁移说明：
  - 本次不需要 SQL 表结构迁移。相关字段均落在现有字符串列或 JSON 文本列中，新增枚举值和返回字段不会破坏历史数据。
  - 旧 `ai_runs.structured_response_json` 保持原样；只有 `prompt_kind = trade-review` 的新记录才按交易复盘结构化字段解析。

## 7. 阶段四：周/月聚合成型

### 7.1 阶段目标

把连续日常记录变成可训练的统计体系。

### 7.2 必做事项

#### A. period_rollups 正式落地

生成：

- Day
- Week
- Month

#### B. trade_metrics 与标签体系

先做最小统计字典：

- pnl_r
- holding_minutes
- result_label
- plan_adherence_score
- ai_alignment_score

配合标签：

- setup
- context
- mistake
- emotion

#### C. 周/月 Review 页面

最少要有：

- KPI 卡
- 盈亏曲线
- setup 排行
- 错误标签排行
- 最佳/最差交易列表

#### D. AI 周/月复盘

基于结构化数据生成：

- 总评
- 优势模式
- 错误模式
- 下阶段行动项

### 7.3 阶段验收标准

1. 能生成一个真实周报
2. 能生成一个真实月报
3. 周/月页中至少有 4 类有效统计
4. AI 周/月复盘不是空文案，而是基于聚合数据输出

### 7.4 推荐并行拆分

- 线程 A：period_rollups 生成器
- 线程 B：trade_metrics 与标签
- 线程 C：Review 页面与图表
- 线程 D：AI 周/月复盘 builder

## 12. 阶段间的决策门槛

不要在主链路没稳时提前推进高级功能。

### 只有满足以下条件，才能进入阶段二点五

- SessionWorkbench 已由真实数据驱动
- 当前上下文与目标挂载稳定
- 删除 / 恢复 / 移动已可用

### 只有满足以下条件，才能进入阶段三

- 连续记录 1 个 Session 没有明显阻塞
- 删除/恢复不破坏数据
- Capture 到事件流回写稳定
- approved 知识卡可稳定检索
- Composer 候选层不会阻塞主流程

### 只有满足以下条件，才能进入阶段四

- 至少已有一批真实 Trade 数据
- TradeDetail 已能稳定显示完整链路
- 交易级复盘已能生成

### 只有满足以下条件，才能进入阶段五

- 周/月聚合已稳定
- 核心统计可信
- UI 主流程没有明显结构性问题

## 13. 下一步最值得新增的并行线程

在 Kickoff 之后，建议改成下面 7 个工作面并行，而不是继续沿用最初那 5 个。

1. `Core Workflow`
- Session
- Current Context
- Event creation

2. `Editor & Blocks`
- 右侧工作台
- 内容块编辑
- 删除/恢复/移动

3. `Capture & Assets`
- 截图
- 标注
- 图片落盘

4. `AI Runtime`
- 真正的 provider 接入
- Prompt builder
- 结构化输出

5. `Trade Loop`
- 开仓到离场闭环
- TradeDetail
- Exit 流程

6. `Review & Aggregation`
- 周/月聚合
- 统计页面
- Review 报告

7. `Knowledge & Composer`
- 知识导入与审核
- Runtime 检索
- Composer 候选
- 锚点记忆

这样拆分更贴近后续真实开发。

## 14. 建议的主控 Prompt 模板

完成 Kickoff 并集成之后，主控线程可以直接使用下面这段提示继续推进：

```text
现在 AlphaNexus 已完成首轮项目启动和基础模块集成。请不要重新搭架子，而是进入 MVP 实施阶段。

实施原则：
1. 以纵向主链路为中心推进：Session -> Screenshot -> Note -> AI -> Trade -> Exit -> Review -> Export。
2. 先打通真实数据链路，再做高级分析和复杂聚合。
3. 所有新增工作必须围绕“可连续盯盘使用”这个目标服务。
4. UI 必须保持干净、清爽、轻量，不要引入噪声型设计。

当前优先级：
P0:
- Current Context 落地
- 截图和标注真落盘
- 右侧工作台可编辑
```

## 15. 最终结论

Kickoff 只是让 AlphaNexus 从 0 到 1 地站起来。  
真正决定项目成败的，是 Kickoff 之后这几轮实施是否足够克制。

正确推进方式不是：

- 先把所有大词做满

1. 先让记录变顺手
2. 再让交易闭环成立
3. 再让周/月洞察可信
4. 最后强化 AI 和体验

如果始终围绕这条路线推进，AlphaNexus 会很快从“好想法”变成“真能每天使用的工具”。
